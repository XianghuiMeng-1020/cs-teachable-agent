# Stage One: Reproducibility Guide

**Document type:** Teacher-facing — how to run the demo and interpret results  
**Date:** 2026-03-12  
**Purpose:** Explain how to run the demo in stub mode and with optional LLM code generation, what environment variables are needed, expected outputs, fallback behavior, and how to interpret the demo.

---

## 1. Prerequisites

- **Python:** Python 3.x (the prototype uses type hints and standard library; no specific minimum version is documented; 3.10+ is typical).
- **Repository layout:** Run from the **repository root** (the folder that contains `prototype/` and `seed/`). Required seed files:
  - `seed/knowledge-units-stage1.json`
  - `seed/sample-problems-stage1.json`
- **Optional (for LLM):** `openai` package (`pip install openai`) and a valid `OPENAI_API_KEY` if you want the TA learner response or LLM-generated code.

---

## 2. How to Run the Demo in Stub Mode (No LLM)

Stub mode uses **only** rule-based behavior: no API calls. The TA learner response and TA code are both from stubs.

**Command (from repository root):**

```bash
python prototype/demo_stage1.py
```

**Or from inside `prototype/`:**

```bash
cd prototype
python demo_stage1.py
```

**Environment variables:** None required. Do **not** set `OPENAI_API_KEY` if you want purely stub behavior for conversation. Do **not** set `USE_LLM_CODE` (or set it to `0` or leave unset) so that Scenario B uses stub code.

**Expected behavior:**

- All three scenarios (A, B, C) run in order.
- Scenario A: Teaching event with only `print_function`; TA learner response (stub); eligible problems = empty; ineligible reasons printed (first 5); selected problem = None.
- Scenario B: Teaching event with `variable_assignment` + `print_function`; TA learner response (stub); selected problem = e.g. `prob_var_001`; TA code = stub correct code; result = PASS; mastery report = proficient.
- Scenario C: Same teaching as B; `force_fail_problem_ids` for `prob_var_001`; TA code = stub wrong code (e.g. prints 21); result = FAIL; mastery report = failing.

The script prints a short warning to stderr if Scenario B does not PASS or Scenario C does not FAIL (e.g. if seed data or code paths change).

---

## 3. How to Run the Demo with Optional LLM Code Generation

To let **Scenario B** use the LLM to generate the TA’s code (still with fallback to stub if the API or output guard fails):

**Environment variables:**

- `OPENAI_API_KEY` — Set to a valid OpenAI API key. Required for any LLM use.
- `USE_LLM_CODE=1` or `USE_LLM_CODE=true` — Enables the LLM code path for Scenario B. Scenario C still uses the stub so the failure path stays deterministic.

**Command (same as above):**

```bash
# Windows (PowerShell)
$env:USE_LLM_CODE="1"
$env:OPENAI_API_KEY="your-key-here"
python prototype/demo_stage1.py

# Unix-like
USE_LLM_CODE=1 OPENAI_API_KEY=your-key-here python prototype/demo_stage1.py
```

**Expected behavior:**

- Scenario A: Unchanged (no problem selected; no code run). TA learner response may come from the LLM if `OPENAI_API_KEY` is set.
- Scenario B: TA code may be **generated by the LLM** instead of the stub. The generated code is checked by the **output guard** (no `def`, `class`, `import`, etc.). If the guard passes and the code runs correctly, result = PASS. If the guard fails or the API fails, the **stub** is used (same as stub mode).
- Scenario C: Always stub wrong code; result = FAIL.

So with LLM code enabled, Scenario B’s code might differ run-to-run (e.g. variable names), but the **outcome** (PASS) is expected as long as the model and guard behave. Any guard or API failure falls back to stub and should still PASS.

---

## 4. Optional: LLM for Conversation Only

If you set `OPENAI_API_KEY` but **do not** set `USE_LLM_CODE`, then:

- **TA learner response** (all three scenarios): May be produced by the LLM (1–2 sentences), constrained by learned units and Stage One scope.
- **TA code:** Still stub only. Scenario B uses stub correct code; Scenario C uses stub wrong code.

So you can test “conversation with LLM, code with stub” without enabling `USE_LLM_CODE`.

---

## 5. Required Environment Variables (Summary)

| Variable | Required? | Effect |
|----------|-----------|--------|
| (none) | No | Demo runs in full stub mode. |
| `OPENAI_API_KEY` | Optional | Enables LLM for TA learner response; required for LLM code when `USE_LLM_CODE` is set. |
| `USE_LLM_CODE` | Optional | Set to `1` or `true` to use LLM for TA code in Scenario B only. Scenario C always stub. |

No other env vars are required for the documented behavior.

---

## 6. Expected Outputs (Stub Mode)

- **Scenario A:**  
  - Teaching event and learned units printed.  
  - TA learner response (one short line).  
  - Eligible problem IDs: empty list or “(none)”.  
  - Ineligible reasons: list of problem_id and missing_units (first 5).  
  - Selected problem: None.  
  - Conclusion line about knowledge state controlling selection.

- **Scenario B:**  
  - Teaching event and learned units.  
  - TA learner response.  
  - Selected problem: e.g. `prob_var_001`.  
  - TA code (multiline).  
  - Result: PASS.  
  - Optional: per–test-case pass/fail.  
  - Mastery report: selected problem, required KUs, learned KUs, Pass/Fail PASS, per-problem interpretation, overall proficient.

- **Scenario C:**  
  - Same teaching as B.  
  - Force-fail set for `prob_var_001`.  
  - Selected problem: `prob_var_001`.  
  - TA code (wrong stub, e.g. `print(age + 1)`).  
  - Result: FAIL (expected).  
  - Mastery report: overall failing.

Final line: “End of demo: A (selection constraint), B (success), C (failure) completed.”

---

## 7. What Fallback Behavior to Expect

- **Conversation:** If `OPENAI_API_KEY` is missing or the OpenAI call fails, the TA learner response is the **stub** (short rule-based sentence). No error is raised; the demo continues.
- **Code (Scenario B):** If `USE_LLM_CODE` is not set, or the LLM returns nothing, or the **output guard** rejects the code (forbidden construct), the **stub correct code** is used. Scenario B should still PASS.
- **Code (Scenario C):** Always stub wrong code; no LLM. Result is always FAIL.

So the demo is designed to **complete successfully** even when no API is configured or when the LLM/guard fails.

---

## 8. How to Interpret the Demo Output

- **Scenario A** confirms that **problem selection is constrained by knowledge state**. With only one unit learned, no problem is selected; the “ineligible reasons” show which units each problem still needs. This supports the design that the test only examines the TA on what it has been taught.

- **Scenario B** confirms the **success path**: teaching two units → a matching problem is selected → the TA produces correct code (stub or LLM) → the code passes the test → mastery is reported as proficient. This shows the full loop: teach → test → report.

- **Scenario C** confirms the **failure path**: same teaching and same problem, but the TA’s code is wrong → the test fails → mastery is reported as failing. This shows that the student (or teacher) can see failure and the TA’s code, supporting reflection and re-teaching.

- **Warnings:** If the script prints “WARNING: Scenario B expected PASS” or “WARNING: Scenario C expected FAIL”, then an assumption of the demo was violated (e.g. seed data change or bug). In normal stub mode, these should not appear.

Together, the three scenarios show that the Stage One prototype implements: zero-knowledge TA, knowledge-state–driven selection, a programming test that examines mastery, and a clear success/failure distinction with mastery reporting—all reproducible with or without an LLM.
