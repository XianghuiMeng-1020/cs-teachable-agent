# Migration: Stage One Prototype to Shared Core

**Document type:** Migration design — bridge from current prototype to full shared-core architecture  
**Governing documents:** MASTER_SYSTEM_BLUEPRINT.md; docs/UNIFIED_KNOWLEDGE_STATE_SCHEMA.md; docs/UNLEARNING_RELEARNING_ENGINE.md; docs/SHARED_CORE_ARCHITECTURE.md; docs/TRACE_AND_HISTORY_LAYER.md  
**Status:** Binding migration plan; implementation must follow this design to preserve claim and reduce drift

---

## 1. Purpose of the Migration Document

This document exists to provide a single, concrete bridge between the current Python Stage One prototype and the full shared-core architecture of the Teachable Agent for CS framework. The governing blueprint, unified knowledge state schema, unlearning/relearning engine, shared core architecture, and trace/history layer are now defined; the prototype is validated and runnable. Without a migration design, the risk is that the prototype remains a one-off Python implementation while new domains or features are built on ad-hoc extensions, or that a future “refactor” rewrites the system from scratch and loses validated behavior. This document explains, in staged and ordered terms, how to move the prototype into the shared-core architecture: what to preserve, what to refactor, what to replace, what to do first and what to defer, and how to do it without breaking the current validated prototype logic prematurely. Migration design is necessary before further implementation or domain expansion so that (1) the blueprint and the codebase stay aligned, (2) the prototype’s proven behavior is retained and extended rather than discarded, and (3) Database and AI Literacy domains are built on a stable, shared foundation rather than on an unstable or unclear one.

---

## 2. What the Current Python Stage One Prototype Already Provides

The current prototype is a real, runnable, documented system. The following capabilities are already in place and are **usable** and **aligned** with future shared-core ideas, though some are still **simplified**.

**Zero-knowledge initialization.** The TA starts with all knowledge units set to `unknown`. StateTracker loads units from the seed file and initializes state to unknown only. This is already usable and aligned: the shared core assumes the same principle (no pre-loaded knowledge except optionally pre-seeded misconceptions). It is simplified in that there is no domain identifier or schema_version in the state yet.

**Structured teaching events.** Teaching is delivered via structured events (topic, knowledge_units_taught, note). `apply_teaching_event()` updates the state tracker deterministically. Already usable and aligned with the “teaching → state update” path. Simplified: no interpretation (correct/incorrect/ambiguous), no routing to a misconception engine, and no teaching_evidence list in state — only status change.

**Constrained problem selection.** Problem selector allows only problems whose required knowledge units are all in the learned set. `get_eligible_problem_ids()` and `get_ineligible_reasons()` implement the shared rule. Already usable and aligned; the only simplification is that “learned” is taken directly from the current thin state (no separate “eligible” policy or partially_learned handling).

**Learner-style response.** TA conversation is generated by `ta_conversation.py` (stub or LLM), constrained by learned units and optional active_misconceptions. Already usable and aligned with the Learner Dialogue Engine role. Simplified: active_misconceptions are passed in but not yet coming from a stateful misconception engine; they are parameters from the demo wiring.

**TA attempt generation.** TA code is produced by `ta_attempt.py` (stub) and `ta_code_generation.py` (LLM with guard and fallback). Generation is constrained by learned units and active misconceptions; output_guard rejects forbidden constructs; fallback to stub on reject or failure. Already usable and aligned with TA Attempt Engine and Guard/Fallback Layer. Simplified: wrong behavior in Scenario C is forced via `force_fail_problem_ids` rather than via state-driven misconception activation.

**Pass/fail evaluation.** Mastery evaluator runs Python code, compares stdout to expected output, and produces pass/fail. Already usable and aligned. Simplified: evaluation adapter is inline (run_python_code) rather than behind a domain adapter interface; results are not yet written into a state-level mastery_history.

**Mastery summary.** Per-unit and overall mastery levels (failing / developing / proficient) are computed from pass/fail using the shared rubric. Already usable and aligned. Simplified: the report is built in memory for the demo; it is not persisted in the knowledge state as mastery_history, and multi-attempt aggregation across runs is not exercised.

**Failure path.** Scenario C demonstrates that the TA can fail a task and that mastery reflects failure. Already usable for teacher-facing demonstration. Simplified: the failure is triggered by a special parameter (force_fail_problem_ids) and fixed wrong stub, not by an active misconception in state, so the misconception lifecycle (activation → behavior → correction → unlearning → relearning) is not demonstrated.

**Optional LLM conversation/code with fallback and guard.** LLM path exists for both dialogue and code; guard rejects forbidden constructs; fallback to stub on reject or API failure. Already usable and aligned with the Guard/Fallback Layer requirement. No simplification here — this is a strength to preserve.

In summary: the prototype already provides zero-knowledge start, structured teaching, state-constrained selection, learner dialogue, TA attempt with guard/fallback, evaluation, and mastery report. These are the right mechanisms; migration extends them into the full schema and shared interfaces rather than replacing them.

## 3. What the Current Prototype Does Not Yet Support

The following gaps exist between the current Stage One prototype and the full shared-core framework. They are stated explicitly so that migration has a clear target.

**Partial or simplified state representation.** The state is a map from unit_id to status (unknown, partially_learned, learned) only. It does not yet include: confidence, active_misconceptions (as a persisted list with activation metadata), teaching_evidence, testing_evidence, correction_evidence, relearning_evidence, mastery_history, or last_updated per unit. There is no domain or schema_version at the global level. The unified schema (UNIFIED_KNOWLEDGE_STATE_SCHEMA.md) requires all of these for evidence tracing and misconception lifecycle.

**No full unified schema fields yet.** Required per-unit fields such as knowledge_unit_name, active_misconceptions (with misconception_id, activated_at, trigger, trigger_reference), and the four evidence lists are absent. Without them, the system cannot record “when was the TA taught,” “when did it fail and with which misconception,” or “when was it corrected and relearned.”

**No full trace/history event writing.** The prototype does not write teaching_event, teaching_interpretation, knowledge_state_update, learner_dialogue, task_selection, ta_attempt, evaluation_result, misconception_activation, correction_event, relearning_event, or mastery_update records. Therefore the chain teaching → state change → behavior → testing → mastery → correction → relearning cannot be reconstructed for evaluation or diagnosis (TRACE_AND_HISTORY_LAYER.md).

**Misconception behavior not yet driven by the shared engine.** Misconceptions exist in the catalog and can be passed into prompts, but there is no Misconception/Unlearning/Relearning Engine. Activation is not triggered by teaching interpretation; correction is not detected; unlearning and relearning are not implemented. Wrong behavior in Scenario C is forced by parameter, not by state. Thus the main claim’s “misconception lifecycle implemented and observable” is not yet met.

**Unlearning/relearning lifecycle not yet operationalized.** The state can move from unknown to learned but cannot move to misconception, then to corrected, then back to learned via relearning evidence. The UNLEARNING_RELEARNING_ENGINE.md logic (activation, correction detection, unlearning transition, relearning evidence, mastery recalculation) is not implemented.

**No persistent history.** State and attempts reset every run. There is no session or run persistence, so mastery_history and evidence cannot accumulate across multiple teaching–testing cycles in a single session or across sessions.

**No multi-attempt mastery aggregation in practice.** The rubric supports pass-rate over multiple attempts, but the demo runs one problem per scenario and does not persist attempts; therefore multi-problem aggregation is not exercised. The shared core expects mastery_history to be updated over time and optionally to weight or exclude attempts during active misconception.

**No domain-general engine boundaries yet.** Code is Python-specific. There is no Domain Layer abstraction (content vs. engine); knowledge units, problems, and misconceptions are loaded directly by the prototype. Adding Database or AI Literacy would currently require copying or forking logic rather than plugging in new content and adapters.

These gaps are not failures of the prototype; they are the expected result of Stage One’s scope. Migration closes them by extending the existing components into the shared architecture and adding the missing engines and trace.

## 4. Migration Principles

The following principles must guide every migration decision. They are non-negotiable for preserving the main claim and avoiding drift.

**Preserve currently validated behavior where possible.** Scenarios A, B, and C, the constrained selection rule, the guard and fallback, and the mastery rubric are already validated. Migration should extend and generalize them, not remove or re-implement them from scratch unless there is a strong reason. When in doubt, add a new path (e.g. “interpreted teaching” path) alongside the existing path rather than replacing the existing path before the new one is proven.

**Migrate incrementally, not by full rewrite.** The migration is staged (see §6). Each stage delivers a coherent step (e.g. state schema expansion, then trace attachment, then misconception engine). A single “rewrite everything” pass is riskier and harder to validate; incremental stages allow regression checks and evidence preservation at each step.

**Move logic into shared-core layers before adding new domains.** Database and AI Literacy must not be started until the Python prototype has been migrated into the shared core (state schema, trace, misconception engine, unlearning/relearning, mastery persistence, and clear Domain Layer boundary). Otherwise new domains will be built on prototype-specific assumptions and will require rework when the core is finally refactored.

**Do not expand UI or deployment while architecture is unstable.** UI polish, hosting, and deployment plans should wait until the shared core (including trace and misconception lifecycle) is in place and the three scenarios still pass. Expanding the surface (new screens, new domains, new infrastructure) before the core is stable increases the cost of later migration and the risk of drift.

**Preserve the knowledge-state-as-single-source-of-truth principle.** No component may determine the TA’s behavior from the LLM, conversation history, or any store other than the knowledge state. During migration, every new read path (e.g. for dialogue or attempt generation) must continue to take its “what does the TA know?” input from the state engine, not from ad-hoc parameters or caches.

**Preserve traceability.** The trace/history layer is not optional. Every migration stage that adds or changes behavior must add or extend trace events so that the full chain remains reconstructable. Evidence for the main claim depends on trace; migration must not leave trace as “later.”

**Do not allow Python-local shortcuts to become permanent framework logic.** Patterns that are acceptable in the prototype (e.g. force_fail_problem_ids, inline evaluation, hard-coded guard rules) must be replaced or generalized so that the shared core has no “if domain == Python” branches for engine behavior. Domain-specificity belongs in the Domain Layer (content and adapters) only.

**Treat migration as claim-preserving architecture work.** The goal is not “clean up the code.” The goal is to elevate the prototype into the architecture that supports the main claim: unified, knowledge-state-constrained, misconception-aware, unlearning/relearning-capable framework. Every migration step should be justifiable in terms of that claim (state schema supports evidence and misconception lifecycle; trace supports evaluation; misconception engine supports observable lifecycle; etc.).

## 5. Prototype-to-Architecture Mapping

This section connects each current prototype component to the target shared-core module and states what is already approximated, what is missing, and whether the component should be refactored, wrapped, replaced, or extended.

**state_tracker.py (StateTracker, load_knowledge_units)**  
- **Maps to:** Knowledge State Engine (partial).  
- **Already approximates:** Per-unit status (unknown, partially_learned, learned); get_state, get_learned_units, update_after_teaching; loading unit definitions from seed.  
- **Missing:** Domain and schema_version; confidence; active_misconceptions; teaching_evidence, testing_evidence, correction_evidence, relearning_evidence; mastery_history; last_updated; APIs for misconception activation, unlearning, relearning.  
- **Action:** **Extend.** Refactor the existing class (or wrap it) so that state conforms to the full schema; add the new fields and transition methods. Do not replace with a new implementation that drops the current behavior; extend the same component.

**teaching_events.py (make_teaching_event, apply_teaching_event)**  
- **Maps to:** Interaction Engine (orchestration) + Knowledge State Engine (state update).  
- **Already approximates:** The “apply teaching → update state” path for correct teaching.  
- **Missing:** Teaching interpretation (correct/incorrect/ambiguous); routing to Misconception Engine when incorrect; recording teaching_evidence in state.  
- **Action:** **Refactor and extend.** Keep apply_teaching_event as the path that the Knowledge State Engine uses for “correct teaching” updates. Add a separate interpretation step (or optional path) that can set misconception_id and call the Misconception Engine; the Interaction Engine (or a thin orchestration layer) will call either “correct path” or “interpreted path” depending on whether interpretation is used. Do not remove apply_teaching_event until the new path is in place and scenarios still pass.

**problem_selector.py (load_problems, select_problem, get_eligible_problem_ids, get_ineligible_reasons)**  
- **Maps to:** Task / Problem Engine.  
- **Already approximates:** Selection rule (required ⊆ learned_units); eligible and ineligible reasons.  
- **Missing:** Input from Knowledge State Engine (eligible units) rather than from a passed-in set; optional domain and task terminology (task_id vs problem_id); trace write for task_selection.  
- **Action:** **Refactor.** Keep the selection logic; change the caller to pass learned_units (or eligible_units) from the Knowledge State Engine. Add trace integration. Optionally rename to task_selector and problem_id → task_id for consistency with shared docs; this is a naming refactor, not a logic change.

**ta_conversation.py (get_ta_learner_response, _fill_prompt, _fallback_stub_response)**  
- **Maps to:** Learner Dialogue Engine.  
- **Already approximates:** State-constrained response (learned_unit_ids, active_misconceptions); stub and LLM paths.  
- **Missing:** Guarantee that input always comes from Knowledge State Engine; active_misconceptions coming from state (not just demo parameters); trace write for learner_dialogue.  
- **Action:** **Refactor.** Ensure the only source of learned units and active misconceptions is the state engine (or a facade that reads from it). Add trace when response is produced. Keep stub and LLM behavior.

**ta_attempt.py and ta_code_generation.py (get_ta_attempt, get_ta_code_attempt, output_guard)**  
- **Maps to:** TA Attempt Engine + Guard / Fallback Layer.  
- **Already approximates:** Stub path; LLM path with guard and fallback; learned units and active misconceptions in prompt.  
- **Missing:** Single “TA Attempt Engine” entry point that orchestrates generate → guard → fallback; guard rules loaded from Domain Layer (optional in Stage One migration); trace write for ta_attempt; wrong behavior driven by state (misconception) instead of force_fail_problem_ids.  
- **Action:** **Refactor and extend.** Treat output_guard as the Guard Layer and the stub in ta_attempt as the Fallback. Expose one interface: “get attempt for task given state.” Add trace (task_id, attempt_id, output summary, guard_passed, fallback_used, active_misconceptions). In a later stage, replace force_fail_problem_ids by ensuring active_misconceptions in state drive the wrong stub or LLM prompt so Scenario C is reproducible via misconception activation.

**mastery_evaluator.py (run_python_code, evaluate_attempt, mastery_summary, build_mastery_report)**  
- **Maps to:** Mastery Evaluator + Domain Layer (evaluation adapter).  
- **Already approximates:** Run code, compare stdout; pass/fail; rubric (failing/developing/proficient); report building.  
- **Missing:** Writing results into Knowledge State Engine (mastery_history); optional policy for weighting attempts during misconception; trace write for evaluation_result and mastery_update; adapter abstraction (so “run and compare” is a Python adapter).  
- **Action:** **Refactor and extend.** Keep run_python_code and evaluate_attempt as the Python evaluation adapter. Have the Mastery Evaluator (shared) call the adapter and then write pass/fail and mastery_history back to the Knowledge State Engine when the state supports it. Add trace. Multi-attempt aggregation is already in the rubric; ensure it is applied when state persists mastery_history across attempts.

**demo_scenarios.py and demo_stage1.py**  
- **Maps to:** Interaction Engine (orchestration).  
- **Already approximates:** The full cycle: teaching → apply_teaching_event → get_ta_learner_response → select_problem → get_ta_code_attempt → evaluate_attempt → build_mastery_report. Scenario A/B/C wiring.  
- **Missing:** Explicit Interaction Engine that calls the same modules in the order defined in SHARED_CORE_ARCHITECTURE §14; trace writes at each step; optional teaching interpretation and Misconception Engine call; no reliance on force_fail_problem_ids for Scenario C once misconception path exists.  
- **Action:** **Refactor.** Replace the inline scenario wiring with an Interaction Engine (or a single “run_cycle” function) that receives teaching event and optional test request, calls Knowledge State Engine, Learner Dialogue Engine, Task Engine, TA Attempt Engine, Mastery Evaluator, and (when present) Misconception Engine in the correct order, and writes trace events. Scenarios A/B/C become “run_cycle with these inputs”; behavior must remain reproducible.

**force_fail_problem_ids (Scenario C)**  
- **Maps to:** Replaced by TA Attempt Engine + state-driven misconception.  
- **Already approximates:** Demonstrates failure path and failing mastery.  
- **Missing:** Failure driven by active misconception in state so that correction and unlearning/relearning can be traced.  
- **Action:** **Replace** in a dedicated stage (after Misconception Engine exists). Scenario C should activate a misconception (e.g. for the unit under test) so that the TA Attempt Engine produces misconception-consistent wrong output; then the same scenario can later demonstrate correction and relearning. Until then, force_fail_problem_ids can remain as a temporary way to run Scenario C; the migration plan must remove it once the misconception path is in place.

**No misconception state or engine**  
- **Maps to:** Misconception / Unlearning / Relearning Engine + Knowledge State Engine (active_misconceptions, correction/relearning evidence).  
- **Missing:** Entire engine: activation on incorrect/ambiguous teaching, correction detection, unlearning transition, relearning evidence and transition. State fields: active_misconceptions, correction_evidence, relearning_evidence.  
- **Action:** **Add.** New module(s) implementing UNLEARNING_RELEARNING_ENGINE.md; state extended to hold active_misconceptions and evidence; Interaction Engine (or teaching pipeline) calls the engine when teaching is interpreted as incorrect or when an explicit correction event occurs.

**No trace/history**  
- **Maps to:** Trace / History Layer.  
- **Missing:** All event types and minimum fields from TRACE_AND_HISTORY_LAYER.md.  
- **Action:** **Add.** Trace layer (log, store, or in-memory for demo) and integration points in each engine or in the Interaction Engine; every event type must be recorded with minimum fields so that the chain is reconstructable.

## 6. Migration Stages

Migration is ordered into stages so that each step is coherent, testable, and dependent on the previous one. Success criteria are stated so that the team can confirm a stage is complete before proceeding.

---

### Stage A: State Schema Expansion

**Goal:** Extend the knowledge state from “unit_id → status” to the full per-unit schema required by UNIFIED_KNOWLEDGE_STATE_SCHEMA.md (confidence, active_misconceptions, teaching_evidence, testing_evidence, correction_evidence, relearning_evidence, mastery_history, last_updated) and add global fields (domain, schema_version). The state becomes the single place that can hold all evidence and misconception data; no new engines are required yet.

**What changes conceptually:** State representation grows to support evidence lists and misconception metadata. Existing APIs (get_learned_units, update_after_teaching) continue to work; new APIs or fields are added for evidence and misconceptions. Domain (e.g. `python`) and schema_version are set so that future multi-domain state is possible.

**What remains unchanged:** Teaching events, problem selection, dialogue, attempt, and evaluation logic continue to use the same inputs and outputs; they may still use only “status” and “learned_units” until later stages. Scenarios A, B, C still run; selection and mastery report behavior are unchanged.

**Evidence of success:** (1) State can be serialized/deserialized with the new fields (defaults for new fields are acceptable). (2) get_learned_units and update_after_teaching still behave as before. (3) All three scenarios still pass (A: no eligible problem; B: pass and proficient; C: fail and failing). (4) No regression in STAGE_ONE_SCENARIO_EVALUATION.md outcomes.

**Why before next stage:** Trace (Stage B) and the Misconception Engine (Stage C) need a state that can store active_misconceptions and evidence; without schema expansion, trace would have nowhere to reflect state_before/state_after and the misconception engine would have no place to write activation or correction data.

---

### Stage B: Trace/History Attachment

**Goal:** Add the Trace/History Layer and integration points so that every relevant event (teaching_event, knowledge_state_update, learner_dialogue, task_selection, ta_attempt, evaluation_result, and later misconception_activation, correction_event, relearning_event, mastery_update) is recorded with the minimum fields defined in TRACE_AND_HISTORY_LAYER.md. Trace can be in-memory or file-based for the prototype; the important outcome is that the full chain is reconstructable.

**What changes conceptually:** After each major action (teaching applied, state updated, dialogue produced, task selected, attempt produced, evaluation completed), a trace record is written. The Interaction Engine (or the current demo wiring) is the natural place to emit or request trace writes so that ordering and trigger links (teaching_event_id, attempt_id) are correct.

**What remains unchanged:** State transitions, selection, dialogue, attempt, and evaluation logic are unchanged. Scenarios A, B, C still run with the same outcomes. No new engines yet.

**Evidence of success:** (1) Running Scenario B produces at least: teaching_event, knowledge_state_update (for the two units taught), learner_dialogue, task_selection, ta_attempt, evaluation_result. (2) From the trace, one can reconstruct “teaching event X led to state update Y; task Z was selected; attempt W was made; evaluation result was pass.” (3) Scenarios A, B, C still pass. (4) Minimum fields per event type are present (event_id, timestamp or sequence_id, domain, and type-specific fields).

**Why before next stage:** The misconception engine and unlearning/relearning (Stages C and D) must write activation, correction, and relearning events to trace; evaluation of the main claim depends on trace. Attaching trace early ensures that no later stage is “retrofitting” trace and that the evidence chain is complete as soon as the misconception lifecycle is implemented.

---

### Stage C: Misconception Engine Integration

**Goal:** Implement the Misconception/Unlearning/Relearning Engine to the extent of **activation only** in this stage: when teaching is interpreted as incorrect (or when an explicit test hook is used), activate a misconception for the affected unit(s) in the knowledge state. The TA Attempt Engine and Learner Dialogue Engine already accept active_misconceptions; they must now receive them from state. Scenario C can be migrated from force_fail_problem_ids to “activate misconception M for the unit(s) tested by the selected problem” so that wrong behavior is state-driven.

**What changes conceptually:** A new module (or set of functions) implements activation: given teaching_event (or a test hook) and misconception_id and unit(s), update state to add the misconception to active_misconceptions and set status to misconception. Teaching interpretation can be minimal at first (e.g. explicit “incorrect teaching” event or a scenario parameter “activate_misconception_id”). The Interaction Engine calls the Misconception Engine when such an event occurs; the Knowledge State Engine (or the same module) applies the state update. Trace: misconception_activation event is written.

**What remains unchanged:** Correct-teaching path is unchanged. Selection, dialogue, attempt, and evaluation still work; when a unit has an active misconception, dialogue and attempt generators already receive it and can reflect it (stub or LLM). Scenario B is unchanged. Scenario A is unchanged.

**Evidence of success:** (1) Scenario C can be run by activating a misconception (e.g. off_by_one_range for for_loop_range) in state instead of using force_fail_problem_ids; the TA still produces wrong code and fails. (2) Trace contains misconception_activation with unit_id, misconception_id, trigger. (3) Optional: removing force_fail_problem_ids from Scenario C and using only misconception activation yields the same failing outcome. (4) Scenarios A and B still pass.

**Why before next stage:** Unlearning and relearning (Stage D) require that misconceptions can be activated and that state has active_misconceptions and correction_evidence; Stage C establishes activation and state shape. Without Stage C, Stage D would have nothing to “unlearn.”

---

### Stage D: Unlearning/Relearning State Transition Support

**Goal:** Implement correction detection, unlearning transition, and relearning evidence and transition per UNLEARNING_RELEARNING_ENGINE.md. When the student (or an explicit correction event) provides corrective teaching, the Misconception Engine removes the misconception from active_misconceptions, sets status to corrected, and writes correction_evidence. When post-correction teaching or a successful task occurs, relearning_evidence is appended; when policy is satisfied, status returns to learned. Mastery recalculation (or weighting) can be applied when state supports mastery_history.

**What changes conceptually:** The Misconception Engine gains correction detection (e.g. explicit correction event or teaching that matches remediation criteria), unlearning (remove from active_misconceptions, set status to corrected, write correction_evidence), and relearning (append relearning_evidence, and when policy is met, set status to learned). The Knowledge State Engine must support these transitions and evidence writes. Trace: correction_event and relearning_event. Optionally, Mastery Evaluator writes mastery_history into state and applies policy for “attempts during misconception” (e.g. exclude or down-weight).

**What remains unchanged:** The correct-teaching path and activation path (Stage C) are unchanged. Selection rule, guard/fallback, and rubric logic are unchanged. Scenarios A and B still pass. Scenario C can be extended to a “Scenario C+” that runs correction and relearning in the same run (teach wrong → activate → fail → correct → relearn → pass or improved mastery).

**Evidence of success:** (1) After activating a misconception and then sending a correction event (or corrective teaching), state shows the unit as corrected (status, empty active_misconceptions, correction_evidence populated). (2) After one or more relearning steps (teaching or successful task), state shows the unit as learned and relearning_evidence populated. (3) Trace contains correction_event and relearning_event with required fields. (4) A full cycle “wrong teaching → activation → failure → correction → unlearning → relearning → learned” can be run and traced. (5) Scenarios A and B still pass.

**Why before next stage:** Mastery aggregation upgrade (Stage E) and shared-core refactor completion (Stage F) assume that state has full evidence and that the misconception lifecycle is in place; evaluation and paper evidence depend on this. Stage D completes the core mechanism that distinguishes the project.

---

### Stage E: Mastery Aggregation Upgrade

**Goal:** Persist mastery in the knowledge state (mastery_history) and ensure multi-attempt aggregation works: multiple tasks or multiple runs (within a session or across persisted state) update mastery_history and the report reflects the shared rubric over all relevant attempts. Optionally, policy for weighting or excluding attempts during active misconception is applied so that “current mastery” reflects post-correction state when relevant.

**What changes conceptually:** Mastery Evaluator writes to the Knowledge State Engine: after each evaluation, append or update mastery_history for the units tested and update overall mastery. The report is built from state (mastery_history and current levels) rather than only from the last attempt. If state is persisted across runs, mastery accumulates; if not, at least within a run (e.g. multiple problems in one scenario) aggregation is correct.

**What remains unchanged:** Rubric (failing/developing/proficient) is unchanged. Pass/fail and evaluation adapter are unchanged. Dialogue, attempt, and misconception lifecycle are unchanged.

**Evidence of success:** (1) After two successful attempts on the same unit (e.g. two different problems), mastery_history shows two entries and the unit’s level is proficient (or as per rubric). (2) After one fail and one pass, level reflects the rubric (e.g. developing). (3) Mastery report is derived from state (mastery_history) and matches the rubric. (4) Scenarios A, B, C (and C+ if implemented) still pass.

**Why before next stage:** Shared-core refactor (Stage F) will formalize the Domain Layer and engine boundaries; mastery must already be “in state” so that the Mastery Evaluator’s contract (read state, evaluate, write back mastery_history) is clear and so that no engine holds mastery in a local structure that is later hard to move.

---

### Stage F: Shared-Core Refactor Completion

**Goal:** Complete the boundary between shared core and Domain Layer: (1) Knowledge State Engine, Interaction Engine, Learner Dialogue Engine, Task Engine, TA Attempt Engine, Mastery Evaluator, Misconception Engine, and Guard/Fallback Layer are clearly separated and use shared interfaces; (2) Domain Layer is a distinct layer that supplies only content (knowledge units, misconception catalog, task bank) and adapters (evaluation adapter, guard rules, fallback stubs); (3) No domain-specific conditionals inside the engines for core behavior (e.g. no “if domain == python” for state transitions or selection rule); (4) Demo/scenario orchestration is the Interaction Engine (single “run_cycle” or equivalent) that calls engines in the order of SHARED_CORE_ARCHITECTURE §14. Optional: rename problem → task, extract guard rules and stubs into a Python domain content package.

**What changes conceptually:** Code is reorganized so that “Python” is one instantiation: Python domain content (seed files) + Python adapters (run code, guard rules, stubs) are loaded by the shared engines. The engines themselves do not branch on domain. The Interaction Engine is the single entry point for “one teaching–testing cycle.” Trace is fully integrated so that every event type is emitted from the right place.

**What remains unchanged:** Behavior of the three scenarios (and C+ if present) is unchanged. The same inputs produce the same outputs; only the organization of code and the clarity of boundaries change.

**Evidence of success:** (1) Adding a second domain (e.g. a minimal “Database” stub with one unit and one task) would require only new content and a new evaluation adapter, not changes to state engine, task engine, or misconception engine logic. (2) Scenarios A, B, C run via the Interaction Engine; no scenario-specific branching inside the engines. (3) Trace contains all event types with minimum fields. (4) Readiness criteria (§11) are met.

**Why before next stage:** Stage G (“ready for domain expansion”) is the declaration that the refactor is complete and that the prototype has been successfully migrated. Stage F is the last technical step; Stage G is the gate.

---

### Stage G: Ready for Domain Expansion

**Goal:** Confirm that the migration is complete and that the Python Stage One prototype is now an instance of the shared-core architecture. No new code is required in this stage; it is a verification and documentation step.

**What changes conceptually:** The team agrees that (1) state schema is full, (2) trace is complete, (3) misconception engine (activation, unlearning, relearning) is operational, (4) mastery is persisted in state and aggregated correctly, (5) shared-core vs. domain boundary is clear, and (6) all readiness criteria in §11 are satisfied. The prototype is no longer “Stage One only”; it is “shared core + Python domain.”

**What remains unchanged:** All behavior and evidence from previous stages.

**Evidence of success:** (1) Readiness criteria (§11) checklist is satisfied. (2) STAGE_ONE_SCENARIO_EVALUATION.md scenarios still pass and are now backed by trace and (where applicable) misconception state. (3) A short migration-complete memo or update to the exit memo states that the prototype has been migrated and that Database/AI Literacy work can proceed on the shared core.

**Why this stage:** Without an explicit “ready for domain expansion” gate, there is a risk of starting Database or AI Literacy on a partially migrated base. Stage G is the decision point: only after it can the team confidently build new domains on the same engines.

## 7. What Must Be Preserved During Migration

The following must not be broken or discarded during migration. Any refactor that undermines these invalidates the migration.

**Zero-knowledge starting point.** The TA must still start with all units unknown (or with only explicitly pre-seeded misconceptions, if that option is added). No pre-loaded “learned” state for normal runs.

**Constrained task selection logic.** The rule “only tasks whose required knowledge units are all in the eligible (learned/partially_learned) set” must remain. No task may be selected when a required unit is missing. Ineligible reasons must still be computable and exposed.

**Reproducible success/failure scenarios.** Scenarios A, B, and C must remain reproducible with the same inputs and the same outcomes: A = no eligible problem; B = pass and proficient; C = fail and failing. After migration, C may be achieved via misconception activation instead of force_fail_problem_ids, but the outcome (TA fails, mastery failing) must be the same.

**Fallback behavior.** When the LLM fails or the guard rejects output, the system must still fall back to the stub path. There must be no path that returns unvalidated LLM output to the student or to the evaluator.

**Learner-style dialogue constraints.** The TA’s conversational response must remain constrained by the knowledge state (learned units, active misconceptions). It must not answer as if it knew concepts it has not been taught; it must reflect misconceptions when they are active.

**The current teacher-facing demo value.** The Streamlit demo (or equivalent) that runs the three scenarios and shows teaching, TA response, problem selection, TA code, grading, and mastery report must continue to work and to demonstrate the same pedagogical points (zero-knowledge TA, constrained selection, success path, failure path).

**Alignment with Stage One evaluation documents.** STAGE_ONE_SCENARIO_EVALUATION.md, mastery-rubric-stage1.md, and the Stage One exit memo describe what the prototype does and what it proves. Migration must not change the observable behavior described there except where explicitly extended (e.g. Scenario C using misconception instead of force_fail). Evidence that “Stage One goals are met” must remain valid.

## 8. What Must Be Replaced or Generalized

The following are temporary or local in the current prototype and must be replaced or generalized so that the shared core does not encode Python-specific shortcuts. This is not a criticism of the prototype; these are normal early-stage simplifications.

**Simplified state tracker.** The current state is unit_id → status only. It must be replaced (by extension) with the full schema: confidence, active_misconceptions, evidence lists, mastery_history, last_updated, domain, schema_version. The replacement is “extend the same component,” not “throw away StateTracker.”

**force_fail style failure simulation.** force_fail_problem_ids is a valid way to demonstrate the failure path in Stage One, but in the full architecture wrong behavior must come from **active misconceptions in state**. So force_fail_problem_ids must be replaced by the path “activate misconception M for unit U → TA Attempt Engine receives active_misconceptions from state → produces wrong output.” Once that path exists and Scenario C runs with it, force_fail_problem_ids can be removed or kept only as a test hook.

**Limited mastery summary.** The report is currently built in memory and not persisted in state. It must be generalized so that mastery is stored in the knowledge state (mastery_history) and the report is derived from state. Multi-attempt aggregation must be supported when state persists across attempts.

**Domain-specific hardcoded behavior.** Guard rules (forbidden constructs), fallback stubs, and the evaluation path (run_python_code) are currently hard-coded in the prototype. They must be generalized so that they are supplied by the Domain Layer (e.g. Python domain content package): guard rules, stub map, and evaluation adapter. The shared Guard/Fallback Layer and Mastery Evaluator call into these adapters without branching on domain name for logic.

**Missing evidence logging.** The prototype does not write trace events. This is not “replaced” but **added**: every relevant event must be written with minimum fields. No component is removed; the trace layer is added and each engine (or the Interaction Engine) contributes its events.

## 9. Order of Refactoring Priority

The migration stages (A → B → C → D → E → F → G) define the order. The rationale is below.

**State schema first (A).** The schema is the contract for all other modules. Trace (B) needs state_before/state_after; the Misconception Engine (C, D) needs active_misconceptions and evidence fields; Mastery (E) needs mastery_history. Without the expanded schema, later stages would either add state fields ad hoc (causing rework) or implement engines that have nowhere to write. So A is first.

**Trace second (B).** Trace is required for evaluation and for the main claim. Attaching trace immediately after the state can represent full updates ensures that every subsequent stage (misconception activation, correction, relearning, mastery writes) is traced from the start. If trace were delayed until after the misconception engine, we would risk forgetting to trace some events or retrofitting trace in a brittle way. So B follows A.

**Misconception engine third (C) and fourth (D).** Activation (C) must come before unlearning/relearning (D) because correction and relearning operate on already-activated misconceptions. C establishes state-driven wrong behavior and replaces the need for force_fail_problem_ids for Scenario C. D completes the lifecycle and enables the full “wrong teaching → correction → relearning” evidence chain. Together they deliver the project’s distinguishing mechanism. So C then D.

**Mastery aggregation fifth (E).** Mastery persistence and multi-attempt aggregation depend on state having mastery_history (from A) and on evaluation results being written back (which can be done as soon as state supports it, but E makes it explicit and ensures the rubric is applied over accumulated attempts). E also allows “mastery after relearning” to be visible. So E after D.

**Shared-core refactor sixth (F).** The refactor that separates Domain Layer from engines and formalizes the Interaction Engine should come after the core behavior (state, trace, misconception, mastery) is in place. Otherwise the refactor would be moving incomplete components. F is the step that makes “adding a new domain = new content + adapters” true. So F after E.

**Ready gate last (G).** Stage G is the verification that A–F are complete and that the system is ready for domain expansion. It has no technical content; it is the checkpoint. So G last.

**Summary priority order:** A (schema) → B (trace) → C (misconception activation) → D (unlearning/relearning) → E (mastery persistence/aggregation) → F (shared-core/Domain Layer boundary) → G (readiness).

## 10. Risks of a Bad Migration Path

The following risks must be acknowledged and mitigated.

**Breaking validated prototype behavior too early.** If migration changes the teaching path, selection rule, or evaluation logic before the new paths are tested, Scenarios A/B/C may fail and the teacher-facing demo value is lost. Mitigation: preserve existing paths; add new paths (e.g. interpreted teaching, misconception activation) alongside; switch over only when the new path is proven and regression tests pass.

**Expanding into new domains before shared-core refactor.** If Database or AI Literacy work starts while the prototype is still Python-specific (no Domain Layer, no shared interfaces), the new domains will duplicate or fork logic. Later refactoring will be costly and the “one framework” claim will be undermined. Mitigation: do not start Database or AI Literacy implementation until Stage F (and G) are complete.

**Adding persistence before schema clarity.** If session or state persistence is added before the state schema is expanded (Stage A), the persisted format will be the thin schema. Migrating persisted data later to the full schema is harder. Mitigation: complete Stage A first; then persistence (if added) persists the full schema.

**Adding UI/deployment before traceability.** If UI or deployment is prioritized before trace is in place (Stage B), runs will not be traceable and the evaluation chain cannot be demonstrated. Mitigation: complete at least Stages A and B before investing in UI or deployment beyond the current Streamlit demo.

**Preserving too much local Python-specific logic.** If migration “wraps” the prototype without generalizing (e.g. guard rules and evaluation stay hard-coded, or the Task Engine still assumes “problems” and “knowledge_units_tested” in a Python-only shape), the shared core will not be truly shared. Mitigation: Stage F explicitly requires Domain Layer extraction and no domain branches in engine logic; treat F as mandatory, not optional.

**Losing the evidence chain needed for the main claim.** If trace is minimal or optional, or if misconception activation/correction/relearning are not traced, the paper cannot demonstrate teaching → state → behavior → mastery → correction → relearning. Mitigation: trace is non-negotiable (Stage B) and every new event type (misconception_activation, correction_event, relearning_event) must be written with minimum fields; use TRACE_AND_HISTORY_LAYER.md as the checklist.

## 11. Readiness Criteria for Post-Migration Architecture

The team can say “the current Python prototype has been successfully migrated into the shared-core architecture” when all of the following are true. These are architecture-level criteria, not deployment or UI criteria.

1. **State schema.** The knowledge state conforms to the full per-unit schema (status, confidence, active_misconceptions, teaching_evidence, testing_evidence, correction_evidence, relearning_evidence, mastery_history, last_updated) and global fields (domain, schema_version). The Knowledge State Engine exposes get_learned_units, get_eligible_units, get_active_misconceptions, and state update APIs for teaching, misconception activation, unlearning, and relearning.

2. **Trace complete.** All event types in TRACE_AND_HISTORY_LAYER.md are recorded with minimum fields: teaching_event, teaching_interpretation (if used), knowledge_state_update, learner_dialogue, task_selection, ta_attempt, evaluation_result, misconception_activation, correction_event, relearning_event, mastery_update. The chain teaching → state change → behavior → testing → mastery → correction → relearning can be reconstructed from the trace.

3. **Misconception lifecycle operational.** Misconception activation (from incorrect/ambiguous teaching or explicit hook), correction detection, unlearning transition (status → corrected, correction_evidence), and relearning (relearning_evidence, status → learned when policy met) are implemented and observable in state and trace. Scenario C (or equivalent) can be run using state-driven misconception activation; optionally, a full cycle including correction and relearning is runnable.

4. **Mastery in state.** Mastery Evaluator writes pass/fail and mastery levels into the knowledge state (mastery_history); the report is derived from state. Multi-attempt aggregation (within a run or across persisted runs) follows the shared rubric.

5. **Shared-core vs. Domain Layer.** Engines (Knowledge State, Interaction, Learner Dialogue, Task, TA Attempt, Mastery Evaluator, Misconception, Guard/Fallback) are clearly separated from the Domain Layer. Domain Layer supplies only content (units, misconceptions, tasks) and adapters (evaluation, guard rules, fallback stubs). No engine contains “if domain == Python” (or equivalent) for core behavior; domain-specificity is only in adapters and content.

6. **Interaction Engine.** A single orchestration entry point (e.g. “run one teaching–testing cycle”) calls the engines in the order defined in SHARED_CORE_ARCHITECTURE §14. Scenarios A, B, C are run by invoking this cycle with the appropriate inputs; no scenario-specific branching inside the engines.

7. **Preserved behavior.** Scenarios A, B, and C produce the same outcomes as in STAGE_ONE_SCENARIO_EVALUATION.md (A: no eligible problem; B: pass, proficient; C: fail, failing). Guard and fallback behavior are unchanged. Zero-knowledge start and constrained selection are unchanged.

8. **Evidence preservation.** The migration does not remove or weaken the evidence that Stage One goals are met; it extends the system so that additional evidence (trace, misconception lifecycle, mastery in state) is available for the full framework claim.

## 12. What Can Start Only After Migration

The following work should wait until migration has reached sufficient maturity (at least Stage F complete and Stage G satisfied). Starting them earlier risks building on an unstable foundation or duplicating logic that will later be refactored.

**Mature Python domain extension.** Expanding the Python domain (more knowledge units, more problems, more misconceptions, natural-language teaching interpretation) should be done on the migrated shared core. The extended Python domain then becomes “shared core + Python content package + Python adapters,” not a larger prototype with its own state and selection logic.

**Database domain implementation.** Database domain (knowledge units, misconception catalog, task bank, query execution adapter, guard rules, fallback stubs) must be implemented only after the shared core is in place. Otherwise Database would be a second codebase with its own state model and selection logic, undermining the “one framework” claim.

**AI Literacy domain implementation.** Same as Database: AI Literacy content and adapters (explanation evaluation, rubric-based scoring) should be added only after the shared core is migrated and the Domain Layer boundary is clear.

**Richer UI.** Beyond the current Streamlit demo (e.g. multi-turn conversation, cross-domain dashboard, richer mastery visualization) should follow migration so that the UI is built on stable engines and trace, not on the pre-migration prototype.

**Persistent analytics.** Session persistence, cross-session mastery, and analytics over trace data (e.g. “how often did correction lead to relearning?”) require that state and trace are in their final shape. These features should be built after Stage E (mastery in state) and Stage B (trace) are complete and, if persistence is added, after the full schema is the persisted format.

**Broader system demonstration.** Demos that show “Python + Database” or “full misconception cycle with correction and relearning” are valuable only when the shared core and at least one full domain (migrated Python) are in place. So broader demonstration follows migration and optionally the first additional domain (Database).

## 13. Migration Guardrails

The following rules are non-negotiable for the migration process.

**Do not re-implement everything from scratch unless necessary.** Prefer extending the existing state tracker, problem selector, mastery evaluator, dialogue and attempt generators, and demo wiring. Replace or rewrite only when the current component cannot be extended without unreasonable complexity (e.g. state tracker might be extended in place; a full rewrite of the state store is not required if extension is feasible).

**Do not allow Python-local shortcuts to become permanent framework logic.** force_fail_problem_ids, hard-coded guard rules, and inline evaluation are acceptable in the prototype. They must not become the long-term design. The migration plan explicitly replaces or generalizes them (Stage C for force_fail; Stage F for guard rules and adapters). Any “temporary” shortcut must have a documented exit (when and how it will be removed or generalized).

**Do not bypass trace/history requirements.** Every new or changed behavior that affects teaching, state, selection, attempt, evaluation, or misconception lifecycle must produce the corresponding trace events with minimum fields. No “we’ll add trace later” for core events.

**Do not let misconception/unlearning/relearning logic remain optional forever.** The blueprint and the main claim require that the misconception lifecycle is implemented and observable. Stages C and D are not optional. Migration is not complete if Scenario C still relies only on force_fail_problem_ids and no correction/relearning cycle is runnable.

**Do not treat migration as mere cleanup.** Migration is claim-preserving architecture work. The goal is to elevate the prototype into the architecture that supports the main claim. Every stage must be justified in terms of that claim (schema for evidence and state; trace for evaluation; misconception engine for lifecycle; mastery in state for reporting; Domain Layer for multi-domain). If a change does not advance the claim or preserve evidence, it is out of scope for this migration.

**Do not start a new domain before readiness (Stage G).** Database and AI Literacy implementation must not begin until the readiness criteria in §11 are satisfied and the team has confirmed that the Python prototype is migrated. This guardrail prevents the “two codebases” or “three pipelines” outcome.

**Do not break the three scenarios.** At the end of every stage, Scenarios A, B, and C must still run and produce the outcomes in STAGE_ONE_SCENARIO_EVALUATION.md. If a stage breaks them, fix before proceeding. Regression is a stop condition.

## 14. Conclusion

This migration document is the real bridge between the Stage One prototype and the full Teachable Agent for CS shared-core architecture. The prototype is valuable and validated; it already implements the right mechanisms (zero-knowledge state, constrained selection, guard/fallback, mastery rubric, learner dialogue, TA attempt). It is also incomplete: state is thin, trace is absent, the misconception lifecycle is not operational, and the boundary between shared core and domain is not yet formalized. Migration elevates the prototype by expanding the state schema, attaching trace, integrating the Misconception Engine (activation, unlearning, relearning), persisting mastery in state, and completing the shared-core refactor so that the same engines and interfaces can support Python today and Database and AI Literacy tomorrow.

Success means: (1) the three scenarios still pass and the teacher-facing demo still demonstrates the same pedagogical points; (2) the knowledge state holds full evidence and misconception data; (3) every relevant event is traced and the chain teaching → state → behavior → mastery → correction → relearning can be reconstructed; (4) wrong behavior can be driven by state (active misconception) and the full correction/relearning cycle can be run and traced; (5) the Domain Layer is distinct and adding a new domain requires only content and adapters; and (6) the team can confidently say that the Python prototype has been migrated into the shared core and that further domain or feature work builds on a stable, claim-preserving foundation.

Immediately after this document, the team should focus on **Stage A: State Schema Expansion** — extending the knowledge state to the full schema without changing behavior — and then **Stage B: Trace/History Attachment** so that every subsequent stage is traced from the start. No new domains, no UI/deployment expansion, and no new features unrelated to the migration should start until Stages A and B are complete and the readiness criteria are within reach.
